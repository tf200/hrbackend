# AGENTS.md

## Overview

This repository is the backend for an HR application written in Go.

The codebase follows a practical clean-architecture style:

- `internal/domain` defines domain models, interfaces, shared errors, and service/repository contracts.
- `internal/repository` implements domain repository interfaces on top of SQLC-generated database queries.
- `internal/service` contains business logic and orchestration.
- `internal/handler` contains Gin HTTP handlers, route registration, request/response DTOs, and mapping between HTTP payloads and domain params.
- `internal/middleware` contains request-scoped middleware such as auth, permissions, request context, and logging.

The runtime is assembled in `internal/app`, and `main.go` is intentionally thin.

## High-Level Flow

The normal request path is:

1. `main.go` loads config and builds the app.
2. `internal/app/app.go` wires DB, logger, token maker, task queue, middleware, repositories, services, and handlers.
3. `internal/handler` registers routes under `/api`.
4. Handlers validate/bind requests and call services using domain params.
5. Services enforce business rules and call repository interfaces from `internal/domain`.
6. Repositories translate between domain types and SQLC types in `internal/repository/db`.

Keep that direction intact:

- handlers depend on services
- services depend on domain interfaces
- repositories implement domain interfaces
- domain should not depend on handler/repository details

## Package Map

### `internal/domain`

This is the source of truth for core contracts.

Put here:

- domain entities and aggregate-shaped structs
- request/result structs shared across layers
- repository interfaces
- service interfaces
- common domain errors
- cross-cutting interfaces such as logger, token, storage, PDF, task queue

Prefer defining new business-facing types here before implementing them elsewhere.

### `internal/handler`

This is the HTTP layer.

Current conventions:

- handlers are grouped by feature, for example `auth_handler.go`, `employee_handler.go`
- routes are sometimes colocated with handlers and sometimes split into `*_routes.go`
- DTOs live in `*_dto.go`
- handlers use `ctx.ShouldBindJSON`, `ctx.Param`, `ctx.Query`, and return JSON through `internal/httpapi`
- handlers map transport types to domain params with small `toXxxParams` and `toXxxResponse` helpers

When adding endpoints:

- keep HTTP-only request/response structs in handler DTO files
- do not leak SQLC types into handlers
- convert authenticated user info from middleware context before calling services
- return consistent envelopes through `httpapi.OK(...)` and `httpapi.Fail(...)`

### `internal/service`

This is where business logic belongs.

Current conventions:

- services are thin structs with domain repository interfaces and a logger
- constructors return domain service interfaces
- validation and business rules happen here before repository calls
- logging happens here for business operations and failures
- services should not know about Gin or raw SQLC models

If logic is reused across services, prefer moving it into `pkg/` or another shared package instead of copy/pasting private helpers.

### `internal/repository`

This layer wraps SQLC.

Current conventions:

- repositories implement interfaces from `internal/domain`
- repositories translate SQLC rows/params into domain structs
- database-specific concerns stay here
- transactions go through `internal/repository/db.Store`

Do not bypass repositories from handlers or services.

### `internal/repository/db`

This contains SQLC-generated code plus transaction helpers in `store.go`.

Important rules:

- files marked `Code generated by sqlc. DO NOT EDIT.` must not be edited manually
- edit SQL in `internal/repository/db/query/*.sql`
- edit schema in `migrations/*.sql`
- regenerate SQLC outputs after query/schema changes

`store.go` also contains custom transactional methods. That file is handwritten and safe to edit.

### `internal/middleware`

Contains auth, permission checks, request ID/context propagation, and request logging.

Auth details are stored on the request context. Use existing helpers such as:

- `AuthPayloadFromContext(...)`
- `EmployeeIDFromContext(...)`
- `RequestIDFromContext(...)`

Prefer propagating identity through context rather than re-reading headers in handlers/services.

### `internal/httpapi`

Contains shared HTTP response primitives like the generic `Envelope[T]`.

Use this package for consistent API responses instead of ad hoc response shapes.

### `internal/seed`

Contains feature-specific seeders and seed state used by `scripts/seed_mock`.

Current seeding model:

- each seeder implements `seed.Seeder`
- `scripts/seed_mock/main.go` owns dependency ordering and transaction boundaries
- seeding runs in a single transaction and commits only after requested seeders finish

If you add seed data for a new feature:

- create a focused seeder under `internal/seed`
- register it in `scripts/seed_mock/main.go`
- define its dependencies explicitly

### `internal/ws`

Contains WebSocket connection/hub code.

This package exists, but it is not yet wired from `internal/app/app.go`. Treat it as isolated infrastructure unless you are explicitly integrating WebSocket support.

### `internal/app`

This is the composition root.

Put here:

- dependency wiring
- runtime adapters
- app lifecycle setup/teardown

Current examples:

- HTTP app build in `internal/app/app.go`
- token adapter in `internal/app/token.go`
- task queue adapter in `internal/app/task_queue.go`

### `pkg`

This is for reusable infrastructure wrappers and utility code that should not live as duplicated private helpers inside services.

Current packages include:

- `pkg/asynq`
- `pkg/jwt`
- `pkg/logger`
- `pkg/password`
- `pkg/bucket`
- `pkg/pdf`
- `pkg/email`
- `pkg/twofa`
- small helpers like `pkg/conv` and `pkg/ptr`

Keep `pkg` generic enough to be reused by multiple features.

## Database and Migrations

Database schema currently lives in:

- `migrations/000001_init.up.sql`
- `migrations/000001_init.down.sql`

This project is still early enough that schema changes are commonly made by editing the initial migration directly.

That is acceptable in this repo right now, but keep these points in mind:

- update SQLC queries if schema changes affect them
- regenerate SQLC after schema/query changes
- keep domain, repository mappings, and handlers in sync with schema changes

Useful migration commands from `justfile`:

- `just up`
- `just down`
- `just force <version>`
- `just up-remote`
- `just down-remote`

## SQLC Workflow

SQLC is configured in `sqlc.yaml`.

When changing persistence behavior:

1. update SQL in `internal/repository/db/query/*.sql` and/or schema in `migrations/*.sql`
2. regenerate SQLC
3. update repository mappings in `internal/repository/*.go`
4. update domain contracts if needed
5. update handlers/services/tests

Do not hand-edit generated files under `internal/repository/db` except the handwritten support files like `store.go`.

## HTTP and Routing Conventions

The API uses Gin and is mounted under `/api`.

Conventions already in use:

- route registration functions are named `RegisterXRoutes`
- handlers are feature structs like `EmployeeHandler`
- auth middleware is applied per route group or per route
- permission checks use string constants inline, for example `EMPLOYEE.VIEW`

When adding a route:

- register it from the feature route file or handler file, following the existing pattern for that feature
- keep permission names consistent with `migrations/permissions_catalog.txt`
- if permissions change, run:
  - `just permissions-check`
  - `just permissions-sync`

## Config

Configuration is loaded through `config/config.go` using Viper from `app.env` and environment variables.

Important values include:

- `DB_SOURCE`
- `SERVER_ADDRESS`
- token secrets and durations
- Redis settings for the task queue
- migration path
- admin seed credentials
- WebSocket ticket TTL / allowed origins

If you add config:

- add it to `config.Config`
- bind it in `LoadConfig`
- validate it in `validateConfig` if it is required

## Background Jobs and Async Work

There is no `internal/worker` package yet.

Current async job support exists through:

- `internal/domain/task_queue.go`
- `internal/app/task_queue.go`
- `pkg/asynq`

If you add new async jobs:

- define the domain payload/interface shape in `internal/domain`
- add adapter wiring in `internal/app` if needed
- implement queue-specific behavior in `pkg/asynq`

If a dedicated worker runtime is introduced later, place it in `internal/worker`.

## Scheduling and OR-Tools

Schedule auto-generation has two implementations:

- stub/default files
- OR-Tools-backed implementation guarded by the `ortools` build tag

Relevant files include:

- `internal/service/schedule_autogen_stub.go`
- `internal/service/schedule_autogen_ortools.go`
- `internal/service/schedule_autogen_common.go`
- `third-party/google-or-tools`

If you touch schedule auto-generation:

- keep stub and OR-Tools paths behaviorally aligned where applicable
- preserve the build tag split
- run the schedule-related tests

## Tests

Tests live next to the code, mostly in `internal/service` and `internal/seed`.

Current testing style favors:

- focused unit tests
- fake repositories instead of full integration setup
- table/data-heavy assertions for business logic

Before finishing non-trivial changes, run targeted tests first, then broader ones if needed.

Useful commands:

- `go test ./...`
- `go test ./internal/service/...`
- `go test ./internal/seed/...`

## Code Style Expectations

When working in this repo:

- keep handlers thin
- keep business rules in services
- keep SQL and SQLC mappings in repositories
- define cross-layer interfaces and structs in `internal/domain`
- prefer explicit mapping helpers instead of passing DB models across layers
- use context-aware logging through the domain logger
- reuse `pkg` helpers instead of duplicating utility code

Avoid:

- putting business logic in handlers
- using SQLC types outside repository and db layers
- bypassing middleware context helpers
- editing generated SQLC files manually

## Practical Change Checklist

For a typical new feature or endpoint:

1. define or extend domain contracts in `internal/domain`
2. add/update SQL in `internal/repository/db/query`
3. regenerate SQLC
4. implement repository methods in `internal/repository`
5. implement business rules in `internal/service`
6. add DTOs/handlers/routes in `internal/handler`
7. add middleware/permissions updates if required
8. add or update seed data if local testing needs it
9. run relevant tests

## Notes for Future Agents

- The package is `internal/handler`, not `controller`.
- `internal/ws` exists.
- `internal/worker` does not exist yet.
- Swagger artifacts are checked into `docs/`, but handler annotations are not widespread yet. Treat docs updates as a deliberate follow-up when changing externally exposed API behavior.
- The worktree may be intentionally dirty. Do not revert user changes unless explicitly asked.
